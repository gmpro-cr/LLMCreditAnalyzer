Problem Statement

A senior banker with 9 years of experience spends 60–70% of their time writing — not thinking. Credit memos are the bottleneck between deal origination and credit committee approval.

In Indian corporate and mid-corporate banking, the credit memo (also called Credit Appraisal Memorandum or CAM) is the primary document through which a banker presents a borrower's creditworthiness to a credit committee. These documents are complex, highly structured, and deeply contextual — yet paradoxically, they are still produced manually, often recycled from previous memos with inconsistent quality.

The current state of credit memo production has three fundamental failure modes:

⏱
Time Intensive
A single mid-corporate CAM takes 8–16 hours to draft from scratch. RMs spend more time writing than managing relationships or originating deals.
📉
Quality Variance
Memo quality depends heavily on the individual RM's experience and writing ability. Junior analysts produce inconsistent, incomplete memos that get rejected at credit.
🔁
Institutional Knowledge Loss
Critical analytical patterns and credit judgments learned over years are locked inside individual bankers' heads, never systematized or shared.
Domain Context
A standard corporate CAM contains: Group overview, Promoter background, Business profile, Industry analysis, Financial analysis (3–5 years), Working capital assessment, Banking arrangement, Proposed structure, Risk identification, and Recommendation. Each section requires synthesis of data from audited financials, projections, field visits, and external research.

User Journey — MVP Flow

01
Case Initiation
RM creates a new case: enters borrower name, CIN/PAN, proposed facility type, and amount. System checks if existing memo history exists for this borrower.
02
Document Upload
RM uploads audited financials (3 years), CMA data, bank statements (6–12 months), CA certificate, and any field visit notes. System accepts PDF, Excel, scanned images.
03
Automated Extraction & Parsing
LLM + OCR pipeline extracts structured data: P&L, Balance Sheet, Cash Flow, banking details. System builds an internal JSON data model of the borrower. Takes ~2–3 minutes.
04
Section-by-Section Generation
System generates all 12 CAM sections sequentially. Each section shows: AI-generated text, confidence level per paragraph, and source citations. RM reviews and edits each section inline.
05
Risk Review & Validation
System presents auto-identified risk flags with severity (High/Medium/Low). RM confirms, adjusts, or adds risks. Low-confidence fields are called out explicitly for verification.
06
Recommendation Drafting
System suggests a credit recommendation structure (approve / approve with conditions / decline) with rationale drawn from the memo. RM writes the final recommendation with AI-generated supporting text as scaffold.
07
Export & Submission
Final memo exported to bank-formatted Word/PDF. System logs RM review confirmation for each section. Version is locked for credit committee. Edits tracked with timestamp and user ID.

Technical Architecture

Frontend
React / Next.js with TailwindCSS. Section-by-section memo editor with inline confidence indicators, edit tracking, and collapsible views. Responsive for desktop (primary) and tablet.
Backend API
Python / FastAPI. Handles document ingestion, LLM orchestration, ratio calculation, and export generation. RESTful endpoints. JWT-based auth. async task queue (Celery + Redis) for heavy extraction jobs.
Document Processing
PDF parsing via pdfplumber + pytesseract for scanned docs. Excel via openpyxl. LLM-assisted extraction for ambiguous/semi-structured tables. Outputs normalized JSON financial model.
LLM Layer
Primary: claude-sonnet-4 or gpt-4o. Section generation via structured prompts with system context (bank, sector, facility type). Confidence scoring via self-evaluation prompting. Fallback to smaller model for ratio narration.
Data & Storage
PostgreSQL for structured borrower data and memo versions. S3-compatible object storage (AWS S3 / Cloudflare R2) for uploaded documents. Redis for session state and task queue.
Security & Compliance
All financial data encrypted at rest (AES-256) and in transit (TLS 1.3). No customer PII sent to third-party LLM APIs unless explicit consent given. Audit log for every AI generation and human edit. Role-based access (RM, Credit Manager, Admin).
Export Engine
Word export via python-docx using bank-specific DOCX templates. PDF via WeasyPrint or headless Chrome. Supports table formatting, headers, footers, and watermarking (DRAFT / CONFIDENTIAL).
Industry KB
Pre-built knowledge base of 25+ sectors in vector store (Chroma / pgvector). Retrieved via semantic search at generation time to inject sector-specific context into industry section prompts.

Human-in-the-Loop & Confidence Design

The most important design principle of CreditGuard AI is that the banker never loses control. Credit decisions carry regulatory, legal, and reputational consequences. The AI is a drafting engine, not a decision engine.

Confidence Level Criteria UI Treatment Required Action
High Extracted directly from structured data (ratio from P&L, bank name from SOA). Deterministic. Green indicator. No warning. Review recommended but not blocked.
Medium Inferred from semi-structured data or interpolated from context. Some judgment applied. Yellow indicator + source citation shown. RM must click "Reviewed" on paragraph before section lock.
Low Generated from weak evidence, assumption-based, or sector KB without specific borrower data. Red indicator + "AI Inference — Verify" badge. Field highlighted. RM must either edit the content or explicitly mark it as verified. Cannot export until resolved.
Design Rationale
Regulatory frameworks (RBI, SEBI) require credit decisions to be made by authorized human officers. The HITL design also protects the bank from model hallucination risk — particularly in sections like Promoter Background and Key Issues where factual accuracy is critical. Every exported memo carries a footer: "AI-assisted draft. Reviewed and approved by: [RM Name, ID, Date]."

Audit Trail Coverage
  • Data Residency: All borrower financial data stored in India (AWS ap-south-1 or equivalent). No customer data leaves Indian jurisdiction without explicit consent.
  • LLM Data Privacy: Financial data sent to LLM APIs only after PII stripping. Option to use self-hosted models (LLaMA / Mistral) for airgapped bank deployments.
  • Concurrent Users: System supports 50 concurrent RM sessions without degradation in MVP; scales to 500 in V2.
  • File Handling: Supports PDF up to 50MB, Excel up to 20MB. Scanned PDFs processed via OCR with accuracy ≥92% for typed text.
  • Browser Support: Chrome, Edge (latest 2 versions). No IE. Mobile read-only; full functionality on desktop/tablet.
